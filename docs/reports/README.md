# 探针报告目录

## 职责

本目录存放项目探针报告（Probe Report）。探针报告是对新技术方案、外部依赖或环境组合的最小实验记录，目的是在正式实施前固化假设、发现约束、降低风险。

探针报告是项目治理的一部分，但不是事实源。事实源是专项计划（`docs/plans/*.md`）和 PLAN_MAP（`docs/PLAN_MAP.md`）。探针报告为计划中的假设和验证方式提供证据基础。

## 触发条件

后续工作满足**任一**条件时，实施前必须先写探针报告：

- 接入外部 API、远端模型服务、HTTP 服务或不稳定第三方能力。
- 尝试新版 MinerU 参数、后端、模型、OCR/VLM 策略或新的环境组合。
- 引入新的输入/输出文件格式、结构化数据格式或图片资源组织方式。
- 改变验证口径、阈值、状态语义、失败处理策略或人工复核策略。
- 需要用真实样本确认性能、内存、准确率、兼容性或幂等性假设。

以下情况**不需要**探针报告：

- 只改错别字、链接、说明文案或治理状态。
- 小范围 bugfix 已有失败复现和明确修复路径。
- 只更新已存在计划中的验收记录，且不改变阶段假设。

## 命名规则

- 使用小写 kebab-case。
- 文件名必须以 `-probe.md` 结尾。
- `<topic>` 应能表达技术对象或实验对象，例如：
  - `mineru-vlm-http-client-probe.md`（MinerU VLM HTTP 客户端探针）
  - `image-path-idempotency-probe.md`（图片路径幂等性探针）
  - `toc-entry-validation-probe.md`（TOC 条目验证探针）

## 最低内容

每份探针报告必须包含以下章节：

1. **背景**：为什么需要这个探针，关联哪个计划或阶段。
2. **问题和假设**：待验证问题、关键假设、不验证范围。
3. **环境和输入**：日期、仓库版本、命令或入口、样本文件、外部服务或依赖。
4. **请求、响应或文件结构**：关键请求参数、响应字段、文件结构、资源路径或错误格式。
5. **最小实验**：可复现命令。
6. **观察结果**：成功路径、失败路径、性能/耗时/内存、输出文件。
7. **约束和风险**：对实施的约束、兼容性风险、回滚方式。
8. **结论**：是否建议进入实施、必须同步到哪个计划、后续验证命令。

## 模板

新探针报告从 [probe-template.md](probe-template.md) 复制，填写各节后存入本目录。

## 与治理的关系

- 探针报告的写入本身不需要治理检查（`check_plan_governance.py` 不检查探针报告存在性）。
- 如果探针报告改变了计划假设、验证方式或完成条件，必须同步对应的专项计划和 PLAN_MAP。
- 探针报告一旦被计划引用为 Step 0 证据，即成为该计划的验证基线的一部分。
