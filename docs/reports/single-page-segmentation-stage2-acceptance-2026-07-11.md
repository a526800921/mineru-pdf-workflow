# 单页分段迁移阶段 2 验收报告

## 结论

**有条件通过，但不能标记为阶段 2 完成。**

备份/恢复机制已实现；单页目录主路径可按页重跑。但完整的页级重跑契约和旧多页兼容验收尚未完成，因此计划状态纠正为 `实施中`，暂不进入阶段 3 代码实施。

## 验收证据

### 页级重跑主路径（有限通过）

- 默认单页目录形如 `p0012-0012`。
- `scripts/pdf-rerun` 传入页码时会定位所属目录；在新的单页目录布局下，起止页相同，因此 MinerU 调用为单页范围。
- 传入旧多页目录（例如 `p0011-0020`）时，仍按旧段范围操作，这是迁移兼容边界。

### 重跑失败恢复

提交 `6783872` 为 `scripts/pdf-rerun` 增加备份/恢复机制：

```text
原始目录 → .backup 临时备份 → 执行重跑
成功     → 删除 .backup
失败     → 删除新目录并恢复 .backup
```

覆盖 MinerU 非零退出、成功但无 Markdown、原始目录存在和原始目录不存在等场景。`.backup` 不匹配正式 `pXXXX-YYYY` 目录名，不干扰 `pdf-merge` 扫描。

### `pdf-auto` 兼容

- `pdf-auto` 仍按 `pXXXX-YYYY` 名称生成 `*-rerun` 目录；单页默认布局下自然退化为页级重跑。
- 重跑失败时不覆盖原始目录，合并继续使用原始结果。
- 旧多页目录仍保留段级重跑行为，满足迁移期兼容。

### 基础验证

```bash
bash -n scripts/pdf-rerun
python3 scripts/check_plan_governance.py .
```

均通过。相关实现提交：`6783872 feat: pdf-rerun 失败保留原始段（备份+恢复机制）`。

## 未完成项

- 旧多页目录下传入单页参数仍会重跑整个所属段；
- `pdf-auto` 仍以段名生成重跑计划，页级行为依赖 `MINERU_SEGMENT_SIZE=1`，没有独立页级 JSON 契约；
- 尚未完成真实 MinerU 服务下的单页成功、无 Markdown、非零退出和恢复原始页端到端复验。

## 修复提交后的再次复核

提交 `6db572c` 已覆盖前一轮发现的主要问题：失败重跑不应用、`no_markdown` 判定、`pdf-auto` 的中间产物同步、`pdf-rerun` 的 v1 content list、残留 backup 处理和 `rerun_detail` JSON 字段。

但仍有以下阻塞项：

1. `content_list_v2.json` 未纳入同步，而页面类型验证依赖 v2。
2. `pdf-rerun` 直接入口没有同步 middle/model/v2 content list/images。
3. 发现旧 `.backup` 时直接删除，原目录缺失时可能丢失唯一备份。
4. `pdf-auto` 覆盖原目录不是事务操作，中间同步失败可能留下半更新结果。
5. 没有独立回归测试覆盖这些失败和恢复场景；当前主要证据来自代码检查和提交说明。

因此修复提交不能直接视为验收通过，阶段 2 仍保持 `实施中`。

## 二次审计新增遗漏

以下问题在首次阶段 2 记录中遗漏，现补充为阻塞项：

1. `pdf-auto` 对 MinerU 非零退出后残留的 Markdown 没有隔离，后续合并循环可能仍会覆盖原始结果。
2. `pdf-auto` 退出码为 0 但无 Markdown 时仍可能把该次重跑视为成功，缺少明确的 `no_markdown` 失败状态。
3. `pdf-auto` 成功覆盖主要只复制 Markdown，不同步 `content_list`、`middle.json`、`model.json` 和图片，存在正文与中间结构不同源风险。
4. `pdf-rerun` 将 Markdown 复制到分段根目录，可能导致 `pdf-merge` 选中根目录文件而找不到同目录 `content_list`，逐页锚点可能丢失。
5. 已存在 `.backup` 时的重跑行为未定义，异常中断后的再次运行缺少安全门禁。
6. JSON 输出没有稳定表达单次页级重跑的实际结果和最终采用结果。

因此当前结论保持：**阶段 2 主路径部分实现，严格验收未通过；阶段 3 暂不实施。**
