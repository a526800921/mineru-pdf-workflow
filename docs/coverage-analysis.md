## 当前覆盖率计算

`scripts/pdf-validate` 当前有两套相关口径：

1. 段级判定：读取 PDF 文本层和分段 Markdown 全文，分词后用 token Counter 计算交集。
2. 逐页详情：如果存在 `content_list_v2.json`，会从结构化内容中提取该页文本，再计算逐页覆盖率。

段级判定流程：

```text
PDF 文本层（PyMuPDF get_text）
        ↓
归一化和分词（中文拆单字，英文数字连续串作为 token）
        ↓
Markdown 全文归一化和分词
        ↓
Counter 多重集合交集
        ↓
coverage = 匹配 token 数 / PDF 总 token 数
```

需要注意：

- 当前 Counter 匹配不关心 token 顺序和位置。
- 标点符号、点号、省略号等会在归一化阶段移除，不会直接成为 token。
- 当前触发 `pdf-auto` 重跑的是段级 `status == suspicious`，逐页 `content_list_v2.json` 详情尚未参与最终重跑决策。

## 低覆盖率的 5 类页面

### 1. 目录页（第 2-8 页，覆盖率 0.4581-0.4630）

**实测证据：** p0001（PDF 第 2 页）覆盖率 0.4581。PDF 文本层 203 token，Markdown 全文 110 token，content_list_v2 提取文本 110 token。Markdown 和 content_list_v2 的 token 数量一致。

PDF 文本层 TOC 页原始形态：

```text
目录
前言 ..............................................................................................................
 .............................................................................................................. 8
重要的注意事项 ...........................................................................................................
 ...........................................................................................................10
```

另一个实测样例显示，目录条目也可能在 MinerU 输出中被部分丢失或重排：

```text
# PDF 文本层
后制动手柄 ................................................................................
  .................................. 34
左手把开关（根据配置）
  ............................................................................................. 35

# MinerU Markdown
.......34

左手把开关（根据配置） ... .......35
```

这个样例中，点线和页码仍然存在，但 `后制动手柄` 这个目录条目标题丢失了；`左手把开关（根据配置）` 则被保留。它说明目录页问题不是单纯的点号处理，而是 **目录条目标题、点线和页码之间的结构关系不稳定**：有些标题会丢失，有些标题会与点线或页码错位。

**根因分析：PyMuPDF 文本层重复提取**

TOC 排版中，每个条目以「标题 + 点线 + 页码」跨多行呈现。PyMuPDF 的 `get_text()` 按视觉布局抽取文本时，同一条目的关键词（如 "前言"、"重要的注意事项"）在标题行和点线行中各出现一次，导致 token 计数翻倍。

对于中文以单字为 token 的验证方式，每次重复都会放大分母而分子不变，覆盖率急剧下降。例如 "车辆视图" 中的 "视" 和 "图" 各出现 5 次（PDF），但 Markdown 中各只出现 1 次。

点号本身会在归一化阶段移除，不影响覆盖率，因此不需归因于省略号或标点 token。核心问题是 **PDF 文本层 token 计数膨胀**，而不是 MinerU 输出缺失。

- **Markdown 输出实际更精确**：去除了 PDF 布局导致的文本重复，结构更清晰。
- 但目录条目标题仍可能真实丢失。例如 `后制动手柄` 整行消失而页码 `34` 残留在上一行末尾；`前言` 条目在 p0001 中也完全丢失，只剩 `.... 8`。
- 目录页对最终正文信息价值较低，但会触发高成本 high 重跑。
- 这类页面适合进入人工复核清单，不需要 high 重跑。

**p0001 实测数据（2026-06-28）：**

| PDF 条目 | MinerU 输出 | 状态 |
|----------|------------|------|
| `前言 .............................................................................................................. 8` | `.... 8` | ❌ 丢失 |
| `重要的注意事项 ...........................................................................................................10` | `重要的注意事项 ............................... ........10` | ⚠️ 保留 |
| `序列号 ......................................................................................................... 12` | `序列号 ........... .... 12` | ⚠️ 保留 |
| `后制动手柄 .................................................................................................................. 34` | （整行消失） | ❌ 丢失 |
| `左手把开关（根据配置） ............................................................................................. 35` | `左手把开关（根据配置） ... .......35` | ⚠️ 保留 |

总计约 18 条目，2 条完全丢失，其余保留但点线破损。p0006（第 7 页）条目基本完整。

### 2. 表格页（第 15-16 页覆盖率 0.8743-0.8917，通过；第 14 页 0.7815 未通过）

实测中第 15-16 页（含表格的正文页）覆盖率超过阈值。第 14 页（p0013）覆盖率 0.7815 未通过，属于混合内容页。

PDF 文本层可能按视觉或行流顺序抽取文本：

```text
发动机 机油 更换 周期 5000km
```

MinerU Markdown 可能输出结构化表格：

```markdown
| 项目 | 周期 |
|------|------|
| 发动机机油 | 5000km |
```

当前段级算法用 Counter 匹配，不关心位置和结构，因此”表格单元格对不上”不是主要问题。表格页更需要确认：

- `content_list_v2.json` 中表格字段是否被当前文本提取逻辑完整抽取。
- PDF 文本层和 MinerU 表格结构化输出是否存在真实文本集合差异（实测中第 15-16 页差异不大）。
- 表格页是否应使用表格专用阈值，或先进入人工复核。

### 3. 图片标签页（第 12-13 页，覆盖率 0.12-0.71）

页面主体是图片，PDF 文本层只有少量标签文字。图片内嵌文字可能由 VLM 或 OCR 识别，但 PDF 文本层并不一定包含这些内容。

这种页面的覆盖率分母和分子都不稳定：低覆盖不一定代表解析失败，高覆盖也不一定代表图片内容完整。更合适的处理方式是标记为 `review_only`。

### 4. 纯文字页（第 9-11、17-20 页，覆盖率 0.85+）

正常段落页覆盖率高，说明当前 token Counter 口径对普通文字页仍然有效。纯文字页低于阈值时，应继续视为可重跑问题。

### 5. 封面页（第 1 页，覆盖率 0.96）

封面文字少且匹配度高。此类页面不是当前优化重点。

## 优化方案

### 方案 A：只降阈值（不建议作为默认方案）

阈值从 0.82 降到 0.6。

- 优点：改动小，能立即减少部分误报。
- 缺点：可能漏过真正解析质量差的纯文字页。
- 适用场景：临时运行参数，不适合作为默认验证策略。

### 方案 B：提升 content_list 口径为主要观测来源

当前 `pdf-validate` 已在逐页详情中读取 `content_list_v2.json`。短期优化不需要完全重写验证器，可以先复用这部分能力：

- 从 `content_list_v2.json` 提取页面级文本、元素类型和结构信息。
- 增加 `page_type`、`decision`、`rerunnable`、`reason` 等字段。
- 保持现有 `coverage` 和 `status` 字段向后兼容。

优点：

- 能直接支撑页面类型识别。
- 对 MCP JSON 契约影响可控。
- 可以先新增观测字段，再逐步改变重跑策略。

风险：

- MinerU 的 `content_list_v2.json` 结构可能随版本变化。
- 需要先用真实样本固定表格、标题、段落、图片字段结构。

### 方案 C：区分可重跑问题和仅需人工复核问题

按页面类型设置处理决策：

| 页面类型 | 初始判断方式 | 默认处理 |
|----------|--------------|----------|
| 文字页 | PDF token 足够多，content_list 主要为文本类元素 | 低于阈值时 `rerun` |
| 表格页 | content_list 含表格结构，或 Markdown 表格占比高 | 初期 `review_only` |
| 目录页 | PDF 文本呈目录条目、页码、点线或短行密集模式 | `review_only` |
| 图片稀疏页 | PDF 文本层 token 很少，content_list 图片元素占比高 | `review_only` |
| 无文本层页 | PDF 文本层为空 | `skip` |

优点：

- 直接减少无效 high 重跑。
- 保留对普通文字页低覆盖的敏感度。
- review.md 可以解释低覆盖原因。

缺点：

- 分类规则需要真实样本校准。
- 表格页初期进入人工复核，不能直接自动判通过。

### 方案 D：目录页条目级验证和修复

目录页不适合继续使用普通 token 覆盖率作为主要质量指标。更合适的做法是从 PDF 文本层抽取目录条目，再检查 MinerU 输出是否保留条目标题。

**条目级验证流程：**

```text
PDF 文本层目录页
        ↓
正则匹配「标题 + 点线 + 页码」模式，抽取候选条目
        ↓
忽略点线、重复点号、换行和页码权重
        ↓
在 Markdown/content_list 中逐条检查标题是否存在
        ↓
输出 toc_entries 清单（含匹配状态和缺失条目）
```

**p0001 实测条目级对照：**

| # | PDF 目录条目 | 页码 | MinerU 中存在 | 缺失详情 |
|---|-------------|------|:---:|------|
| 1 | 前言 | 8 | ❌ | 仅剩 `.... 8`，标题丢失 |
| 2 | 重要的注意事项 | 10 | ⚠️ | 标题保留，点线断裂 |
| 3 | 序列号 | 12 | ⚠️ | 标题保留 |
| 4 | 参数 | 13 | ⚠️ | 标题保留 |
| ... | ... | ... | ... | ... |
| 17 | 后制动手柄 | 34 | ❌ | 整行消失 |
| 18 | 左手把开关（根据配置） | 35 | ⚠️ | 标题保留 |

条目级结果更精确：人工复核从”整页覆盖率 0.46 太低”收敛到”2/18 条目丢失，其余可接受”。

**候选 `toc_entries` JSON 字段：**

```json
{
  “page”: 1,
  “page_type”: “toc”,
  “toc_entries”: [
    {“title”: “前言”, “page_ref”: 8, “found”: false, “match_text”: null},
    {“title”: “重要的注意事项”, “page_ref”: 10, “found”: true, “match_text”: “重要的注意事项 ....10”},
    {“title”: “后制动手柄”, “page_ref”: 34, “found”: false, “match_text”: null}
  ],
  “toc_stats”: {“total”: 18, “found”: 16, “missing”: 2, “partial”: 0}
}
```

字段语义：

| 字段 | 含义 |
|------|------|
| `title` | 从 PDF 文本层抽取的目录条目标题 |
| `page_ref` | 目录条目指向的页码 |
| `found` | 标题是否在 MinerU 输出中找到 |
| `match_text` | 匹配到的文本片段，未找到时为 null |
| `toc_stats` | 汇总计数，包含 total/found/missing/partial |

**后续修复流程：**

如果 TOC 缺失条目较多（如 >20%），可触发后处理修复：

1. 从 PDF 文本层抽取所有目录条目标题和页码。
2. 生成规范 Markdown 列表，例如 `- [前言](#) 8`。
3. 仅对被识别为 `page_type: toc` 的段执行替换或附加。

修复策略选项：
- **替换模式**：用 PDF 抽取的条目列表完全替换 MinerU 的 TOC 输出（最干净，但丢失任何格式）
- **补全模式**：保留 MinerU 输出中已存在的条目，仅补全丢失的条目（更安全）

该修复应作为后续增强，改变最终 Markdown 内容前需验证跨行目录、多级目录和多列目录场景。

### 方案 E：句子级或段落级检索比较（长期）

不拆单字，而是按完整句子或段落检查：

1. 从 PDF 文本层提取句子或段落。
2. 在 MinerU 输出中检索是否出现。
3. 允许有限模糊匹配。
4. coverage = 找到的句子数 / 总句子数。

优点：

- 更接近人工判断。
- 对普通正文的误报可能更少。

缺点：

- 实现复杂。
- 模糊匹配阈值不好校准。
- 表格、目录、图片页仍需要分类策略。

## 建议路径

1. **✅ 立即**：修正文档认知，不把目录页问题归因于省略号 token，也不把表格问题归因于 token 位置。（已完成，2026-06-28）
2. **短期**：在 `pdf-validate` 中新增页面类型和处理决策字段，但先保持旧 `status` 兼容。
3. **短期**：让 `pdf-auto` 只重跑 `rerunnable == true` 的段；目录页、图片稀疏页和初期表格页进入 `review_only`。
4. **中期**：增加目录页条目级验证，忽略点线和页码权重，输出具体缺失目录条目。
5. **中期**：基于真实 `content_list_v2.json` 样本完善结构化文本提取，尤其是表格字段。
6. **长期**：评估目录页后处理修复，以及句子级或段落级检索比较。

专项实施计划见 [覆盖率验证口径优化](plans/coverage-validation-optimization.md)。
