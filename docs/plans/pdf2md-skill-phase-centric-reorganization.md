# 计划：pdf2md skill 阶段中心化重排

## 计划状态

- 状态：已完成
- 当前阶段：阶段 3：独立可读性验收
- 最后更新：2026-07-17

本文档是对已完成 `pdf2md-skill-sequential-workflow` 的后续增量计划。前一计划建立了阶段 0～9 导航；本计划解决阶段导航与详细操作说明分离的问题，将现有事实归位到对应阶段。

## 背景与问题

当前 `skills/pdf2md/SKILL.md` 的阶段 0～9 集中在文件前部，后续大量内容仍按主题平行展开。LLM 需要在阶段入口和后文之间来回拼接同一阶段的工具、输入、输出、人工确认和失败边界，容易漏掉门禁或错误跳阶段。

## 目标

- 让每个阶段成为一个完整执行单元：目标、进入条件、操作顺序、tool、输入、输出、人工确认、成功条件和失败处理集中在一起。
- 将 ModelPad、输出包、页面分类、人工修复、TOC/页码、结构化抽取、审核、入库和下游交付分别归入对应阶段。
- 保留必要的全局不变量，但删除与阶段流程重复的第二套“核心流程”。
- 保持现有 CLI、输出包结构、Schema、状态门禁、动态脚本安全边界和 CLI-only 决策不变。
- 同步项目级和用户级 `pdf2md` skill，并通过独立可读性检查。

## 非目标

- 不修改代码、CLI、公共 JSON/CSV Schema 或真实 PDF 输出包。
- 不改变任何阶段的业务行为、页码门禁、审核规则或下游交付契约。
- 不新增 MCP Server、数据库导入能力或第二个 skill 入口。
- 不把详细契约复制到多个阶段；同一事实只保留一个位置。

### 内容归位原则

- 阶段说明是唯一主流程；后续不再保留与阶段平行的“核心流程”或“统一顺序”章节。
- 阶段 0～3 承载输入、服务、解析、质量分类和按页证据读取。
- 阶段 4～5 承载人工修复、动态脚本、TOC/页码和 canonical 发布单元同步。
- 阶段 6～8 承载抽取配置、LLM/用户审核、入库前状态和导出门禁。
- 阶段 9 承载 chunks、`downstream_delivery.md` 和下游消费边界。
- 全局只保留触发描述、事实源/同步规则、禁止事项和“原始证据只读”等跨阶段不变量。

## 风险与回滚

- 归位时遗漏某个公共契约：用现有主题章节关键词和下游契约反向搜索；失败则恢复 skill。
- 阶段正文过长仍难以使用：将低频详细字段移入阶段末尾的“本阶段契约”，但不再建立平行主流程。
- 项目级和用户级 skill 漂移：同步后强制 `cmp`，不一致时不推进完成。
