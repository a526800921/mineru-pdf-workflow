# ADR 0003：LLM 编排与受控动态辅助脚本

## 状态

已接受

## 背景

本项目的实际使用方式是：用户提供 PDF、确认 PDF 中的事实和审批结构化候选，LLM 负责分析问题、编排工具和维护产物。现有 CLI 已覆盖主要流程，但真实 PDF 仍可能出现某个现有脚本无法直接处理的一次性问题。

如果遇到脚本缺口就立即修改通用脚本，会把单个 PDF 的特殊语义带入公共实现；如果要求用户自己编写或执行脚本，又不符合当前协作方式。

## 决策

### 1. 用户入口继续使用 `pdf2md` skill

- 用户只需要与 `pdf2md` skill 对话，不需要记忆 CLI 参数。
- `pdf2md` skill 负责识别当前阶段、读取产物、决定下一步、向用户展示需要确认的事实，并编排 CLI。
- `pdf2md-fix` 的人工协作能力已经迁入 `pdf2md`；用户已批准废弃旧兼容 skill，后续只维护 `pdf2md` 入口。历史计划和产物中的 `manual_fixes.jsonl` 等名称仅表示修复记录，不表示仍存在第二个 skill 入口。

### 2. CLI 继续作为确定性执行层

- 解析、合并、校验、按页修复、抽取、审核状态计算和批次导出继续由 `scripts/` 执行。
- CLI 不承担某个 PDF 的业务语义判断，不自动批准 `needs_review`，不把 VLM 输出直接当最终事实。
- 项目继续遵循 ADR 0002 的 CLI-only 决策，不新增 MCP Server 或 MCP 兼容层。

### 3. 允许 LLM 生成受控动态辅助脚本

当现有脚本无法安全完成一个明确、有限、可验证的操作时，LLM 可以生成一次性或包级辅助脚本，但必须满足：

- 优先组合现有 CLI；只有组合不足时才生成辅助脚本。
- 执行前备份目标 Markdown、manifest、相关 JSON/CSV 和修复记录，并记录备份 hash。
- 先 dry-run，输出将修改的文件、页码/record_id、前后 hash 和变更摘要。
- 限制修改范围，只允许修改用户明确授权的派生产物；原始 PDF、segments、content_list 不可改写。
- 具备页锚点、record_id、来源 hash 或等价的目标校验。
- 具备幂等性；重复运行不能重复写入或扩大范围。
- 执行失败时整组回滚，不能只恢复 Markdown 而遗漏 manifest 或配置。
- apply 完成后必须执行显式、只读、可审计的验证命令；验证失败或验证阶段写入包内文件时，整组回滚。
- 动态辅助脚本不能授权修改 `review_overrides.csv`、`ingest_ready.csv`、`conflicts.csv`、`ingest_batch.jsonl`、`ingest_manifest.json` 等审批或入库前门禁产物；这些状态只能由既有审核/导出流程在用户确认后更新。
- 一次性脚本默认放在临时目录；只有需要复现或再次使用时，才登记为包内辅助脚本并保留命令、输入、输出和 hash。

### 4. 动态脚本的晋升规则

- 只服务单个 PDF、单次异常：保留为临时辅助脚本或包内配置，不进入通用脚本。
- 同类问题在多个 PDF 重复出现：先补充最小回归测试，再将逻辑晋升为通用 CLI。
- 晋升后的通用逻辑必须通过 GitNexus 影响分析、回归测试、`pdf-check-fixes` 和治理文档复核。

## 为什么当前不需要 MCP

当前调用方和脚本在同一工作区，LLM 已经能够调用本地命令、读取文件和处理结果。MCP 在此处只会增加一层工具协议、参数映射、服务生命周期和兼容性维护，不能减少用户的语义确认工作。

只有出现以下需求时，才重新评估 MCP 或其他远程接口：

- PDF 处理服务必须被多个外部客户端远程发现和调用；
- 需要跨机器权限、队列、异步任务和统一任务状态；
- CLI 无法满足外部系统的稳定契约，且已有明确的服务边界和运维责任。

在这些条件出现前，继续采用“skill 入口 + CLI 执行层 + 用户审批 + 受控动态脚本”的方式。

## 后果

优点：

- 用户不需要学习脚本。
- 一次性异常不会污染通用代码。
- 通用问题仍有清晰的晋升路径。
- 备份、hash、dry-run 和回滚保留可审计性。

代价：

- LLM 需要承担更多流程编排和变更前检查。
- 动态脚本必须严格区分临时工具、包内配置和通用实现。
- 没有 MCP 的远程发现能力，外部系统必须直接调用 CLI 或另建集成计划。
